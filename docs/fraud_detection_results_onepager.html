On the “Same AWS VPC” Concern

“Same VPC” prevents public internet routing — it does not eliminate service-hop latency. A DynamoDB read from within the same AWS VPC still invokes a managed service endpoint over HTTP (p99 under load: 10–30ms). A SageMaker endpoint invocation adds another 5–20ms of service-plane overhead before model execution begins. Per-hop cost exists regardless of VPC boundary.

  • SPCS runs on AWS EC2 in the same AWS region as Zilch’s infrastructure. The physical network is identical.
  • AWS PrivateLink connects Zilch’s VPC directly to Snowflake’s VPC without the public internet — the same networking primitive AWS uses for its own managed service endpoints. The ~3–8ms ingress hop is comparable to an EKS → DynamoDB or EKS → SageMaker call in Zilch’s current stack.
  • Fewer hops at equivalent per-hop cost is structurally faster. Zilch’s current path: EHI → API Gateway → Kinesis → SageMaker → DynamoDB → Triton (~5 service hops). Snowflake path: EHI → PrivateLink → SPCS → OFS internal mesh (~2 hops). ~14ms p99 is the measured result.
